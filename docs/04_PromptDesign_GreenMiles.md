## 1. 문서 개요

본 문서는 그린마일즈(GreenMiles) 프로젝트를 다른 AI 또는 Codex와 같은 코딩 에이전트에게 단계별로 위임할 수 있도록, 그대로 복사해 사용 가능한 고품질 프롬프트 모음을 제공한다. 모든 프롬프트는 GreenMiles 프로젝트 맥락(PRD/TRD/User Flow)을 기준으로 작성되었으며, 코드 샘플은 포함하지 않는다. 다만 파일 경로, 기능명, 테스트 기준 등은 구체적으로 명시한다.

---

## 2. Prompt Design 원칙

| 원칙 | 설명 |
|---|---|
| 맥락 우선 | 모든 프롬프트는 GreenMiles의 프로젝트 정의(PRD/TRD/User Flow)를 명시적으로 포함한다 |
| 범위 명확화 | MVP 범위와 제외 범위를 프롬프트 안에 명시하여 과도한 기능 확장을 방지한다 |
| 권한 명시 | Codex에게 허용되는 작업과 금지되는 작업을 명확히 구분한다 |
| 완료 기준 포함 | 모든 작업 프롬프트는 "완료 기준"을 포함하여 검증 가능하게 한다 |
| 법적 표현 통제 | "탄소배출권 지급 완료"가 아닌 "예상 탄소 기여도", "GreenMiles 포인트" 등 단계별 표현을 강제한다 |
| 코드 샘플 금지 | 기획/설계 프롬프트에는 코드 샘플을 요청하지 않는다(개발 프롬프트는 코드 작성을 지시하되 본 문서 자체에는 코드 예시를 포함하지 않는다) |

## 3. 공통 프롬프트 템플릿

아래 템플릿을 모든 세부 프롬프트의 기본 골격으로 사용한다.

```
역할: [AI에게 부여할 역할]
프로젝트 맥락: GreenMiles는 저탄소/로컬 농식품의 생산-유통-소비 데이터를 연결하여
소비자에게는 GreenMiles 리워드 포인트를, 유통사에는 ESG 데이터를 제공하는
블록체인 기반 농식품 탄소 리워드 플랫폼이다. (PRD/TRD 요약 첨부)
작업 목표: [구체적으로 수행할 작업]
입력 데이터: [참고할 문서/데이터 목록]
제약 조건: [MVP 범위, 금지 표현, 기술 스택 제약 등]
출력 형식: [Markdown/표/목록 등]
검증 기준: [작업 완료를 판단할 기준]
```

이 템플릿은 마크다운 코드 블록이 아닌 일반 텍스트 형태로 각 세부 프롬프트에 적용된다.

## 4. PRD 개선용 프롬프트

역할: 너는 농식품 ESG/기후테크 분야의 시니어 프로덕트 매니저다.
프로젝트 맥락: GreenMiles PRD(첨부)를 검토 대상으로 한다.
작업 목표: 현재 PRD의 비즈니스 모델, 성공 지표, 리스크 섹션을 더 구체화하고, 유통사 SaaS 구독료의 가격 구조 가설을 추가하라.
입력 데이터: PRD: GreenMiles 문서 전체
제약 조건: 법정 탄소배출권 발행을 전제하지 말 것, MVP 범위(가상 POS/QR/영수증)를 벗어나는 기능을 핵심 지표에 포함하지 말 것
출력 형식: 기존 PRD 구조를 유지한 Markdown, 변경 사항은 표로 비교 정리
검증 기준: 모든 신규 주장에 `[AI 보완]` 또는 `[추정]` 표시가 되어 있는지 확인

---

## 5. TRD 개선용 프롬프트

역할: 너는 Supabase/Next.js/블록체인 테스트넷 경험이 있는 시니어 백엔드 아키텍트다.
프로젝트 맥락: GreenMiles TRD(첨부)의 16개 테이블 설계와 권한 구조를 검토 대상으로 한다.
작업 목표: RLS 정책의 구체적인 조건문 수준의 설명(코드 제외)과 reward_rules의 조건 스키마 예시 구조를 보강하라.
입력 데이터: TRD: GreenMiles 문서 전체
제약 조건: 실제 SQL/코드는 작성하지 말 것, 테이블/필드명은 기존 명명 규칙을 유지할 것
출력 형식: 표 중심 Markdown
검증 기준: 기존 16개 테이블과의 정합성, 신규 필드 추가 시 타입과 설명 포함 여부

## 6. UI/UX 설계용 프롬프트

역할: 너는 클린테크/그린 파이낸스 서비스를 다수 설계한 프로덕트 디자이너다.
프로젝트 맥락: GreenMiles는 소비자에게는 "장보기 리워드" 중심, 유통사에게는 "데이터 중심 대시보드"를 제공해야 한다.
작업 목표: 소비자 지갑 화면과 유통사 ESG 대시보드의 정보 구조(IA)와 핵심 컴포넌트 목록을 설계하라.
입력 데이터: User Flow: GreenMiles 문서, PRD의 9번(화면 분위기), 11번(UI/UX 참고) 항목
제약 조건: VCM/MRV/Scope 3 같은 전문 용어는 툴팁 또는 상태 배지로 단순화할 것, 소비자 화면은 모바일 우선, 유통사 화면은 데스크톱 우선
출력 형식: 화면별 컴포넌트 목록 표, 정보 우선순위 목록
검증 기준: User Flow 문서의 단계와 화면 요소가 1:1로 매핑되는지 확인

---

## 7. 데이터베이스 설계용 프롬프트

역할: 너는 Postgres/PostGIS/Supabase RLS에 능숙한 데이터베이스 설계자다.
프로젝트 맥락: GreenMiles TRD의 16개 테이블(profiles ~ mrv_batch_items)을 기준으로 한다.
작업 목표: 각 테이블 간 외래키 관계도를 텍스트 기반으로 정리하고, RLS 정책 초안을 역할별로 자연어로 작성하라.
입력 데이터: TRD: GreenMiles 8번(주요 테이블 설계), 9번(역할 기반 권한 설계) 섹션
제약 조건: 실제 SQL 코드는 작성하지 말 것, 개인정보·구매 상세는 온체인 저장 대상에서 제외할 것
출력 형식: 관계 설명 목록 + 역할별 권한 표
검증 기준: consumer/farmer/retailer/admin/mrv_partner 5개 역할 모두에 대한 정책이 포함되었는지 확인

## 8. 리워드 알고리즘 설계용 프롬프트

역할: 너는 탄소회계(Carbon Accounting) 및 인센티브 설계 전문가다.
프로젝트 맥락: GreenMiles는 운송거리, 저탄소 인증 여부, 생산 방식, 배출계수를 결합해 리워드를 산정한다.
작업 목표: 각 변수별 가중치 산정 기준안을 `[AI 보완]` 표시와 함께 제안하고, 결정적(deterministic) 계산이 가능한 산식 구조를 자연어로 설명하라.
입력 데이터: TRD: GreenMiles 10번(리워드 계산 로직), 11번(푸드 마일리지 기반 인센티브 구조) 섹션
제약 조건: 실제 탄소배출권 발행과 연결하지 말 것, 코드/수식 코드블록 대신 자연어 또는 표로 설명할 것
출력 형식: 변수별 가중치 표, 산정 절차 단계 목록
검증 기준: 동일 입력값에 대해 항상 동일한 결과가 나오는 구조인지(결정성) 설명 포함 여부

## 9. 유통사 ESG 대시보드 설계용 프롬프트

역할: 너는 B2B SaaS 대시보드 설계 경험이 많은 프로덕트 디자이너 겸 데이터 분석가다.
프로젝트 맥락: 유통사는 판매량, 리워드 발행량, 재구매율, 로컬푸드 전환율, Scope 3 참고 데이터를 확인해야 한다.
작업 목표: 대시보드의 핵심 위젯 구성과 필터(기간/매장/SKU/카테고리) 우선순위를 설계하라.
입력 데이터: TRD: GreenMiles 15번(유통사 ESG 대시보드 데이터 집계 방식), PRD의 9번 화면 분위기
제약 조건: Scope 3 데이터는 "참고용"임을 화면에 명시할 것, 공식 인증 데이터처럼 표현하지 말 것
출력 형식: 위젯별 목적/데이터소스 표
검증 기준: 모든 위젯이 TRD의 집계 항목과 1:1로 매핑되는지 확인

## 10. MRV/블록체인 설계용 프롬프트

역할: 너는 MRV(측정·보고·검증) 프로세스와 블록체인 앵커링 구조에 능숙한 기후테크 엔지니어다.
프로젝트 맥락: GreenMiles는 검증 대상 리워드 이벤트를 MRV 배치로 묶고, 배치 해시를 테스트넷에 기록한다.
작업 목표: MRV 배치 생성부터 블록체인 앵커링까지의 상태 전이(state transition)를 자연어로 상세히 설명하라.
입력 데이터: TRD: GreenMiles 16번(MRV batch 생성 방식), 17번(블록체인 해시 앵커링 방식)
제약 조건: 메인넷 배포, 실제 토큰 발행/거래는 다루지 말 것, Solidity 코드는 작성하지 말 것
출력 형식: 상태 전이 표(draft → submitted → verified → anchored / rejected)
검증 기준: 모든 상태 전이에 대해 실패 시 처리 방식이 함께 설명되었는지 확인

## 11. QA/테스트 케이스 생성용 프롬프트

역할: 너는 핀테크/기후테크 서비스 QA 경험이 풍부한 QA 엔지니어다.
프로젝트 맥락: GreenMiles MVP의 핵심 루프는 "구매 인증 → 리워드 계산 → 지갑 반영 → 대시보드 확인"이다.
작업 목표: User Flow 문서의 정상 플로우와 예외 플로우(QR 만료, 중복 claim, 영수증 검증 실패, SKU 누락, MRV 반려, 블록체인 기록 실패)에 대한 테스트 케이스를 작성하라.
입력 데이터: User Flow: GreenMiles 문서 전체
제약 조건: 테스트 코드가 아닌 테스트 케이스 명세(시나리오/기대 결과)로 작성할 것
출력 형식: 테스트 케이스 표(ID/시나리오/사전조건/기대결과)
검증 기준: User Flow의 모든 예외 상황 6가지가 각각 최소 1개 이상의 테스트 케이스로 커버되는지 확인

## 12. Codex 개발 시작용 프롬프트

역할: 너는 GreenMiles 프로젝트의 풀스택 개발을 담당하는 시니어 엔지니어다.
프로젝트 맥락: GreenMiles는 저탄소/로컬 농식품 구매 데이터를 QR/영수증으로 인증하고, 푸드 마일리지 기반 리워드를 제공하며, 유통사에는 ESG 대시보드를, 검증 데이터는 MRV 배치 후 블록체인 테스트넷에 해시로 기록하는 플랫폼이다.
작업 목표: 프로젝트 초기 세팅 — Next.js App Router + TypeScript + Tailwind + shadcn/ui 프로젝트를 생성하고, Supabase 프로젝트 연결 구조와 폴더 구조(`/app`, `/components`, `/lib`, `/types` 등 경로 예시)를 설계하라.
입력 데이터: TRD: GreenMiles 전체, 특히 4번(권장 기술 스택), 5번(프론트엔드 구조), 6번(백엔드 구조)
권한: 파일 생성·수정·삭제, 패키지 설치, 터미널 명령 실행, 테스트 실행, 브랜치 생성, PR 제안은 허용한다. 단, production 배포, mainnet 배포, 실제 API 비용 발생, private key 또는 secret commit, 개인정보 외부 전송은 명시적 승인 전 금지한다.
완료 기준: 프로젝트가 로컬에서 정상 빌드/실행되고, 폴더 구조와 환경변수 템플릿(`.env.example`, 실제 값 제외)이 README에 문서화되어 있을 것

## 13. Codex 기능별 작업 프롬프트

### 13.1 1차: 프로젝트 세팅 / DB / Auth / RLS

역할: 시니어 풀스택 엔지니어
작업 목표: TRD의 16개 테이블(profiles, organizations, organization_members, farms, stores, product_skus, emission_factors, reward_rules, pos_events, purchase_items, receipt_submissions, wallets, reward_events, ledger_entries, mrv_batches, mrv_batch_items)에 대한 Supabase 스키마 마이그레이션과 Supabase Auth 연동, 역할별 RLS 정책을 구현하라.
권한: 파일 생성·수정·삭제, 패키지 설치, 터미널 명령 실행, 테스트 실행, 브랜치 생성, PR 제안은 허용한다. 단, production 배포, mainnet 배포, 실제 API 비용 발생, private key 또는 secret commit, 개인정보 외부 전송은 명시적 승인 전 금지한다.
완료 기준: 모든 테이블이 마이그레이션으로 생성되고, 역할별 RLS 테스트(consumer/farmer/retailer/admin/mrv_partner)가 통과할 것
테스트 기준: 각 역할이 본인 데이터 외 접근 시 거부되는지 검증하는 테스트 작성

### 13.2 2차: 가상 POS / QR claim / 리워드 엔진

역할: 시니어 풀스택 엔지니어
작업 목표: 가상 POS 이벤트 생성 기능, QR claim token 발급/검증 기능, 푸드 마일리지 기반 리워드 계산 엔진(결정적 로직)을 구현하라.
권한: 1차와 동일한 권한 문구 적용
완료 기준: QR claim 시 중복/만료 검증이 정상 동작하고, 동일 입력에 대해 항상 동일한 리워드 결과가 산출될 것
테스트 기준: 중복 claim 시도, 만료된 토큰 claim 시도에 대한 테스트 케이스 통과

### 13.3 3차: 소비자 지갑 / 유통사 대시보드

역할: 프론트엔드 + 백엔드 엔지니어
작업 목표: 소비자 지갑 화면(적립 포인트/구매 이력/예상 탄소 기여도/MRV 상태)과 유통사 ESG 대시보드(필터/판매량/재구매율/Scope 3 참고 데이터)를 구현하라.
권한: 1차와 동일한 권한 문구 적용
완료 기준: User Flow 문서의 8번, 9번 플로우가 화면에서 동일하게 동작할 것
테스트 기준: 필터 변경 시 집계 데이터가 정확히 갱신되는지 검증

### 13.4 4차: 영수증 업로드 / 관리자 검토

역할: 풀스택 엔지니어
작업 목표: 영수증 업로드, (선택적) OCR 보조 추출, 관리자 검토/승인/반려 화면을 구현하라.
권한: 1차와 동일한 권한 문구 적용
완료 기준: 승인된 영수증이 purchase_item으로 전환되어 리워드 계산까지 연결될 것
테스트 기준: 반려 시 소비자에게 사유가 정확히 전달되는지 검증

### 13.5 5차: MRV batch / 블록체인 앵커링

역할: 백엔드 + 블록체인 엔지니어
작업 목표: MRV 배치 생성, 상태 전이(draft → submitted → verified → anchored), 테스트넷 해시 앵커링 기능을 구현하라.
권한: 1차와 동일한 권한 문구 적용. 추가로: 반드시 테스트넷(Polygon Amoy 등) 기준으로만 작업하며, 메인넷 관련 설정은 일체 추가하지 말 것.
완료 기준: 배치 생성부터 테스트넷 앵커링까지 전체 흐름이 데모 가능한 수준으로 동작할 것
테스트 기준: 앵커링 실패 시 재시도 큐 동작 여부 검증

### 13.6 6차: 테스트 / 데모 / README 정리

역할: QA 엔지니어 + 테크 리드
작업 목표: 전체 핵심 루프에 대한 통합 테스트 작성, 데모 시나리오 스크립트 작성, README에 실행 방법과 아키텍처 요약을 정리하라.
권한: 1차와 동일한 권한 문구 적용
완료 기준: 신규 개발자가 README만으로 로컬 환경을 재현할 수 있을 것
테스트 기준: 핵심 루프(구매 인증 → 리워드 → 지갑 → 대시보드 → MRV → 앵커링) 통합 테스트 전체 통과
